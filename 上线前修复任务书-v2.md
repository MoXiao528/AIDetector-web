# 上线前修复任务书 v2

更新时间：2026-03-31（第二轮修复后）

## 0. 本版约束

这版任务书按当前真实上线目标收口，不做超范围扩展。

- 首页单文件拖拽/上传：**不强制登录**
- 多文件批量上传：**当前不作为线上可见功能**
- 支付/订阅：**当前不做**
- 联系页：**可以保留**
- 后端测试环境：使用 `conda activate lab`

## 1. 结论

现在距离“能上线”不是差一堆大功能，主要差的是：

1. 部署面还有硬风险
2. 首页上传链路和当前权限模型不一致
3. 标色预览对复杂结构内容会破坏原格式
4. 认证和敏感数据处理还没收紧
5. contract 和运行时还没完全对齐

其中真正的上线阻塞，我按优先级排成：

- `P0`：部署安全、首页上传链路、复杂排版标色预览
- `P1`：认证限流、递归脱敏、契约同步
- `P2`：上传文件保留原始排版、后续结构化导入增强

## 2. 不建议现在上多 Agent

当前阶段先**不需要**多 Agent。

原因：

- 现在首先要定边界，不是并行堆改动
- 标色预览修复会同时碰前端状态流、结果映射、历史记录展示，主线程必须先把方案定死
- contract 同步也不能抢跑，不然容易和实现互相打架

等真正开始改代码时，可以拆成 3 路：

1. 后端安全与鉴权
2. 前端上传与标色预览
3. contract 与测试回归

但不是现在。

### P0-2 首页单文件上传链路改为“游客可用”

目标：首页单文件上传继续可用，但不再依赖登录态。

你的要求已经明确：

- 首页拖拽/上传不要强制登录
- 多文件批量上传当前不开放

当前问题：

- 首页上传走的是后端 `/api/v1/detect/parse-files`
- 这个接口要求 `ActiveMemberDep`
- 结果就是首页看起来能上传，实际匿名用户会撞 401

推荐方案：

**不要给首页单文件上传硬塞登录。**

直接改成和 Dashboard 一样的客户端解析路径：

1. 单文件上传统一走前端 `readTextFromFile`
2. 解析完成后写入 `scanStore`
3. 后续检测仍然走现有 guest scan
4. 多文件上传继续隐藏，不开放入口

为什么这样改：

- 你的首页上传本质上只是“把文件内容塞进编辑区”
- 这件事没必要走一个登录保护的后端解析接口
- 这样最省事，也最符合你现在的产品设定

验收标准：

- 未登录用户首页上传 `txt/docx/pdf` 不会 401
- 上传后文本能进入编辑区并继续检测
- 当前线上没有任何地方暴露多文件批量上传入口

涉及文件：

- `D:\Code\AIDetector-web\src\sections\HeroSection.vue`
- `D:\Code\AIDetector-web\src\utils\fileReaders.js`
- `D:\Code\AIDetector-web\src\api\modules\scan.ts`
- `D:\Code\AIDetector-Back\backend\app\api\v1\detections.py`

备注：

- 后端 `parse-files` 接口可以先保留，后续如果做会员能力或批量导入再用
- 当前首页不应该继续依赖它

### P0-3 修复“复杂排版内容标色预览会打平格式”

目标：检测后的预览必须保留原始排版结构，不能把复杂内容压成纯文本段落。

当前现象：

- 纯段落文本正常
- 标题、列表、换行、强调、复杂结构内容会被打平

当前根因链：

1. 编辑器里保留的是 `editorHtml`
2. 但检测时主要发送的是 `inputText`
3. `inputText` 是通过 `extractTextFromHtml(editorHtml)` 提取出来的纯文本
4. 检测结果回来后，预览 HTML 又会被 `buildHighlightedHtml(text, sentences)` 这种逻辑重建
5. 重建过程使用的是统一的 `<p><span>...</span></p>` 模板，不是原始 DOM
6. 后端返回的 `highlighted_html` 也是按纯文本分段拼出来的

这意味着：

- 现在的标色预览，不是“在原文上高亮”
- 而是“拿原文文本重新拼一份高亮 HTML”
- 所以复杂结构一定丢

### 这项修复的上线版目标

上线前先修到这个程度：

1. **对编辑器内输入/粘贴的富文本内容**，检测后预览保留原始结构
2. 高亮逻辑基于原始 `editorHtml` 做结构映射，不再把它降级成纯文本再重建
3. 历史记录里如果有 `editor_html`，同样优先基于它渲染预览

### 这项修复不应做成什么

不要继续把下面这些当主路径：

- `plainTextToHtml(text)`
- `buildHighlightedHtml(text, sentences)`
- 后端直接下发一整块打平后的 `highlighted_html` 当唯一显示源

### 推荐修复方向

上线前版本建议这样做：

1. 检测请求时，把当前 `editorHtml` 一并保存在前端流程中
2. 检测结果返回后，优先基于 `editorHtml` 做高亮渲染
3. 高亮粒度先按**段落/块级映射**做，不要一上来做字符级 DOM surgery
4. 对命中的块节点加高亮 class 和 `data-sentence-id`
5. 保留原始块结构、列表结构、换行和内联强调标签

为什么先做块级映射：

- 当前后端给的是段落范围 `start_paragraph / end_paragraph`
- 不是字符 offset
- 所以上线前最稳的办法是“保留 DOM 结构 + 块级染色”
- 这已经能解决你现在说的“复杂结构被打平”

### 单独说明：上传文件的原始格式保留，不在本轮完全解决

这个要单独说清楚：

- 前端 `docx/pdf` 读取现在本来就只提取纯文本
- 后端 `parse-files` 也只返回 `content`
- 所以**如果输入来源本身就是文件上传，目前没有足够信息恢复原始排版**

也就是说：

- 本轮可以修好“编辑器富文本内容检测后预览不打平”
- 但如果你要“上传 docx/pdf 后还能在预览里保留原文结构”，那是下一层增强，不是这轮上线修补能顺手带掉的

验收标准：

- 标题、列表、换行、加粗文本在检测后预览仍保留结构
- 高亮只改变命中区域样式，不改变文档排版骨架
- 历史记录预览和当前结果预览表现一致
- 对普通纯文本内容不引入回归

### P2-1 文件上传保留结构化排版

这项不是当前上线阻塞，但如果你要把平台往学术写作工作台走，这是必做项。

当前问题：

- `docx` 前端读法是 `convertToHtml -> div.textContent`
- `pdf` 读法是提取文字后拼接
- 后端 `parse-files` 也只返回纯文本 `content`

所以上传文件后想保留标题、列表、强调结构，当前根本没有原始结构可用。

后续正确方向：

1. `docx` 保留结构化 HTML 或中间 AST
2. `pdf` 至少保留 block 级布局信息
3. 上传接口返回 `content + html/blocks`
4. 预览渲染基于结构化内容而不是纯文本

这项不建议插到当前上线修复里，否则节奏会失控。
